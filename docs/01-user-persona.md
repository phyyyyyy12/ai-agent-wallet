# AI Agent 钱包 — 用户画像与使用场景

## 1. 概述

AI Agent 钱包服务于两类角色：**人类用户**（钱包的拥有者/管理者）和 **AI Agent**（钱包的直接操作者）。钱包需要同时满足人类对资产安全和可控性的要求，以及 Agent 对编程化接口和自动化操作的需求。

---

## 2. 人类用户画像（钱包拥有者/管理者）

### 2.1 Web3 开发者

| 维度 | 描述 |
|------|------|
| **背景** | 熟悉区块链技术，日常使用 MetaMask 等钱包 |
| **动机** | 让 AI Agent 代替自己执行重复性链上操作 |
| **核心需求** | 编程友好的接口、详细的操作日志、可配置的权限边界 |
| **痛点** | 手动操作效率低，但又不放心完全放手给 Agent |

### 2.2 DeFi 策略执行者

| 维度 | 描述 |
|------|------|
| **背景** | 有一定的金融和链上交易经验 |
| **动机** | 让 Agent 7×24 自动执行链上操作 |
| **核心需求** | 支出限额保护、操作可审计 |
| **痛点** | 人不可能 24 小时盯盘，但 Agent 自主交易金额过大有风险 |

---

## 3. AI Agent 画像（钱包直接操作者）

### 3.1 Agent 的特性

| 特性 | 说明 |
|------|------|
| **无 UI 交互** | 通过 MCP 工具调用钱包，不使用图形界面 |
| **7×24 运行** | 可持续执行，不需要休息 |
| **上下文有限** | 每次对话有 context 限制，需要持久化状态 |
| **可被操控** | 存在 prompt injection 风险，可能被诱导执行恶意交易 |

### 3.2 Agent 的约束

- **不具备"常识"判断**：无法判断一笔交易是否"合理"，需要人类预设策略
- **不能保管私钥**：私钥不应暴露给 Agent 的上下文，钱包应封装签名逻辑
- **操作需可审计**：每一步操作都需记录，方便事后追溯

---

## 4. 使用场景

### 场景 1：自动支付链上服务费用

> Agent 在执行任务时需要调用某个链上服务，自动支付所需费用。

- 触发：Agent 判断需要调用付费服务
- 钱包行为：检查余额 → 校验限额 → 签名并发送交易
- 安全约束：单笔限额 + 日累计限额

### 场景 2：链上数据查询

> Agent 查询某地址余额、交易记录，用于分析决策。

- 触发：Agent 需要链上信息
- 钱包行为：调用只读方法，无需签名
- 安全约束：无限制（只读操作）

### 场景 3：Agent 身份验证

> Agent 需要向某个服务证明"我就是 `0x1234...` 这个地址的控制者"，而不发起任何转账。

- 触发：第三方服务要求链上身份验证（如 Web3 登录、API 鉴权）
- 钱包行为：用私钥对指定消息签名 → 返回签名结果
- 安全约束：纯签名操作，不消耗 Gas，不涉及转账

### 场景 4：跨 Agent 协作转账

> 多个 Agent 协作完成任务，其中一个 Agent 需要将资金转给另一个 Agent 的钱包。

- 触发：协作流程中的资金调度
- 钱包行为：校验目标地址 → 限额检查 → 执行转账
- 安全约束：白名单机制限定可转入地址

---

## 5. 总结

AI Agent 钱包的核心设计原则：

1. **人类掌控**：人类设定策略边界，Agent 在边界内自主操作
2. **最小权限**：Agent 只能执行被授权的操作类型和金额范围
3. **完全透明**：所有操作可审计、可追溯
4. **安全优先**：私钥隔离、限额保护、频率熔断
