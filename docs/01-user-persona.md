# AI Agent 钱包 — 用户画像与使用场景

## 1. 概述

AI Agent 钱包服务于两类角色：**人类用户**（钱包的拥有者/管理者）和 **AI Agent**（钱包的直接操作者）。人类用户涵盖三类画像，钱包需要同时满足人类对资产安全和可控性的要求，以及 Agent 对编程化接口和自动化操作的需求。

---

## 2. 人类用户画像（钱包拥有者）

### 2.1 普通区块链持有者

| 维度 | 描述 |
|------|------|
| **背景** | 持有少量加密资产，不懂技术，没有使用过 MetaMask 等专业钱包 |
| **动机** | 想对自己的资产做基本操作：转账、查余额、看记录 |
| **核心需求** | 用自然语言描述操作意图，不需要理解地址格式、Gas 等概念 |
| **痛点** | 链上操作门槛高，地址填错不可逆，Gas 费不知道怎么设 |

### 2.2 接受加密支付的自由职业者

| 维度 | 描述 |
|------|------|
| **背景** | 设计师、程序员等，客户用加密货币付款，需要频繁收款、查记录、偶尔转账 |
| **动机** | 像用微信收款一样简单地管理加密资产，不想学钱包技术 |
| **核心需求** | 自然语言操作、随时查流水、转账有记录可对账 |
| **痛点** | 传统钱包门槛高，每次操作都要查教程，客户催款时手忙脚乱 |

### 2.3 需要 Agent 代操作的小型机构/团队

| 维度 | 描述 |
|------|------|
| **背景** | Web3 项目方、小型基金、协议运营团队，链上资金操作频繁，开始引入 AI Agent 提效 |
| **动机** | 让 Agent 承担日常链上操作，同时保留完整的操作记录以备内部审查 |
| **核心需求** | 私钥不暴露给 Agent、每笔操作可追溯、超限交易须人工介入 |
| **痛点** | 直接给 Agent 私钥风险不可控；出了问题无法追溯是人操作还是 Agent 操作；缺乏轻量级的资金安全边界 |

---

## 3. AI Agent 画像（钱包直接操作者）

### 3.1 Agent 的特性

| 特性 | 说明 |
|------|------|
| **无 UI 交互** | 通过 MCP 工具调用钱包，不使用图形界面 |
| **7×24 运行** | 可持续执行，不需要休息 |
| **上下文有限** | 每次对话有 context 限制，需要持久化状态 |
| **可被操控** | 存在 prompt injection 风险，可能被诱导执行恶意交易 |

### 3.2 Agent 的约束

- **不具备"常识"判断**：无法判断一笔交易是否"合理"，需要人类预设策略
- **不能保管私钥**：私钥不应暴露给 Agent 的上下文，钱包应封装签名逻辑
- **操作需可审计**：每一步操作都需记录，方便事后追溯

## 4. 使用场景

### 场景 1：自然语言转账

> 用户用自然语言描述转账意图，Agent 解析并执行链上转账。

- 触发：用户输入如"给 Alice 转 0.1 ETH"或"把 50 美元的 ETH 转到 0x1234..."
- 钱包行为：解析意图（地址/金额/币种）→ 检查余额 → 校验限额 → 签名并发送交易
- 安全约束：单笔限额 + 日累计限额

### 场景 2：链上数据查询

> Agent 查询某地址余额、交易记录，用于分析决策。

- 触发：Agent 需要链上信息
- 钱包行为：调用只读方法，无需签名
- 安全约束：无限制（只读操作）

### 场景 3：Agent 身份验证

> Agent 需要向某个服务证明"我就是 `0x1234...` 这个地址的控制者"，而不发起任何转账。

- 触发：第三方服务要求链上身份验证（如 Web3 登录、API 鉴权）
- 钱包行为：用私钥对指定消息签名 → 返回签名结果
- 安全约束：纯签名操作，不消耗 Gas，不涉及转账


---

## 5. 总结

AI Agent 钱包的核心设计原则：

1. **人类掌控**：人类设定策略边界，Agent 在边界内自主操作
2. **最小权限**：Agent 只能执行被授权的操作类型和金额范围
3. **完全透明**：所有操作可审计、可追溯
4. **安全优先**：私钥隔离、限额保护、频率熔断
