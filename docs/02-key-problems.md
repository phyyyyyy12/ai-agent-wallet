# AI Agent 钱包 — 重点问题分析

在设计 AI Agent 钱包时，需要重点解决以下 3 个核心问题。

---

## 问题 1：私钥安全 — 如何在 Agent 环境中安全管理密钥？

### 问题描述

传统钱包的私钥由人类在安全设备（硬件钱包、手机）中保管。但 AI Agent 运行在服务器/本地进程中，私钥需要被程序访问来签名交易。这引入了多个攻击面：

- **私钥泄露**：如果私钥以明文存在于内存或磁盘，Agent 被入侵后私钥直接暴露
- **Prompt Injection**：恶意输入可能诱导 Agent 输出私钥或签署恶意交易
- **上下文泄露**：Agent 的对话日志可能被记录，私钥不应出现在任何上下文中

### 解决方案

| 策略 | 实现方式 |
|------|----------|
| **加密存储** | 私钥使用 AES-256 加密后存储为 keystore 文件，密码不与 Agent 共享 |
| **签名隔离** | Agent 通过 MCP Tool 调用签名，私钥在钱包内部解密和使用，不暴露给 Agent 上下文 |
| **权限分离** | keystore 文件权限设为 600，仅钱包进程可读 |

### 本项目的实现

- 使用 `cryptography` 库的 Fernet 对称加密存储私钥
- MCP Server 作为独立进程运行，Agent 通过工具调用间接使用签名能力
- 私钥永远不出现在 MCP 的输入/输出中

---

## 问题 2：自主决策的边界 — Agent 能做什么、不能做什么？

### 问题描述

AI Agent 的核心价值是"自主执行"，但在金融场景中无限制的自主权是危险的：

- Agent 可能因为"幻觉"而向错误地址转账
- Agent 可能被 prompt injection 诱导执行大额转账
- Agent 没有"这笔交易是否合理"的常识判断能力
- 不同场景需要不同的权限粒度

### 解决方案：分级授权策略引擎

```
操作分级:
┌─────────────────────────────────────────┐
│  Level 0: 只读操作                       │  ← Agent 自由执行
│  查余额、查交易、查合约状态               │
├─────────────────────────────────────────┤
│  Level 1: 小额操作                       │  ← Agent 自主执行（受限额约束）
│  小额转账、Gas 支付                      │
├─────────────────────────────────────────┤
│  Level 2: 大额/敏感操作                  │  ← 需要人类审批
│  大额转账、合约交互、白名单外地址         │
├─────────────────────────────────────────┤
│  Level 3: 管理操作                       │  ← 仅人类可执行
│  修改限额、导出私钥、更换钱包             │
└─────────────────────────────────────────┘
```

### 具体策略

| 策略 | 说明 | 默认值 |
|------|------|--------|
| **单笔限额** | 单次转账金额上限 | 0.1 ETH |
| **日累计限额** | 24小时内总支出上限 | 1.0 ETH |
| **地址白名单** | 仅允许向已知安全地址转账 | 空（即所有地址需限额内） |
| **频率限制** | 每分钟最大交易数 | 5 笔 |

### 本项目的实现

- `SecurityPolicy` 类管理所有策略配置
- 每笔交易前经过 `check_transaction()` 校验
- 超出策略的交易返回 `NEEDS_APPROVAL` 状态而非拒绝，留给人类决定

---

## 问题 3：可观测性 — 如何知道 Agent 做了什么？

### 问题描述

AI Agent 是"黑盒"运行的，人类无法实时观察 Agent 的每一步操作。在涉及真金白银的场景中：

- 人类需要知道 Agent 发起了哪些交易
- 出现异常时需要快速定位和回溯
- 需要审计证据来确保 Agent 按策略执行
- 需要实时感知 Agent 的操作状态

### 解决方案

| 层面 | 实现 |
|------|------|
| **操作日志** | 每次 MCP Tool 调用记录：时间、工具名、参数、结果、耗时 |
| **交易历史** | 链上交易记录持久化：txHash、from、to、value、status、blockNumber |
| **安全事件** | 限额触发、审批请求、异常交易单独标记和记录 |
| **前端仪表盘** | React Dashboard 实时展示余额、交易历史、Agent 操作日志、安全策略状态 |

### 日志格式示例

```json
{
  "timestamp": "2026-04-20T10:30:00Z",
  "tool": "send_eth",
  "params": {"to": "0x1234...", "amount": "0.05"},
  "security_check": "PASSED",
  "tx_hash": "0xabcd...",
  "status": "SUCCESS",
  "gas_used": 21000,
  "duration_ms": 3200
}
```

### 本项目的实现

- 所有操作写入 JSON Lines 格式的日志文件
- FastAPI 提供 `/api/logs` 和 `/api/transactions` 接口
- React Dashboard 展示实时数据

---

## 总结

| 问题 | 核心矛盾 | 解决思路 |
|------|----------|----------|
| 私钥安全 | Agent 需要签名能力 vs 私钥不能暴露 | 签名隔离 + 加密存储 |
| 决策边界 | Agent 需要自主权 vs 无限制自主权危险 | 分级授权 + 策略引擎 |
| 可观测性 | Agent 黑盒运行 vs 人类需要掌控全局 | 全量日志 + 实时仪表盘 |
