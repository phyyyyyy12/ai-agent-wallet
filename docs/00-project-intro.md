# AI Agent 钱包 — 项目完整介绍

---

## 一、项目背景

随着 AI Agent 技术的快速发展，越来越多的智能体开始参与到自动化任务中——包括调用 API、处理数据、甚至执行链上操作。然而，现有的加密货币钱包（如 MetaMask）都是为人类设计的，需要图形界面点击确认，AI Agent 根本无法直接使用。

**那么，如何为 AI Agent 设计一个专属的加密货币钱包？** 这就是本项目要解决的问题。

---

## 二、核心洞察：双层用户模型

设计 AI Agent 钱包，首先要想清楚"用户是谁"。

这里有一个容易踩的坑：很多人会说"用户就是 AI Agent"。但这个答案是不完整的。

**AI Agent 钱包实际上服务于两类用户：**

- **人类用户（Owner）**：钱包的真正拥有者，负责设定策略、监控资产、审批大额操作。他们是 Web3 开发者、DeFi 策略执行者等。
- **AI Agent（Operator）**：钱包的实际操作者，通过程序接口发起转账、查询余额、签名消息。它没有界面，7×24 运行，但不能完全自主。

这个双层模型决定了整个架构的设计方向：**人类设定边界，Agent 在边界内自主执行。**

## 三、核心要解决的三个问题

### 问题 1：私钥安全

AI Agent 运行在服务器进程中，如果私钥以明文存储，一旦 Agent 被恶意 Prompt 操控，攻击者可能直接拿走私钥。

**解决方案：签名隔离架构**
- 私钥使用 AES-256 加密存储在本地 keystore 文件中
- Agent 调用签名工具时，私钥在钱包进程内部完成签名，**结果返回给 Agent，私钥本身永远不出现在 Agent 的上下文中**
- 即使 Agent 被 Prompt Injection 攻击，攻击者也无法获取私钥

### 问题 2：自主决策的边界

AI Agent 没有"这笔交易是否合理"的常识判断能力，可能因为幻觉或被操控而发起危险交易。

**解决方案：分级授权策略引擎**

```
Level 0：只读操作       → Agent 自由执行（查余额、查交易）
Level 1：小额转账       → Agent 自动执行（受限额约束）
Level 2：大额/陌生地址  → 需要人类审批（NEEDS_APPROVAL）
Level 3：管理操作       → 仅人类可执行
```

类比银行：就像银行的常用收款人名单和每日转账限额——白名单内的地址、限额内的金额 Agent 直接转；超出范围的需要人类确认。

安全检查分三个结果：
- `APPROVED`：自动通过
- `NEEDS_APPROVAL`：需要人类确认
- `DENIED`：直接拒绝（触发频率风控）

### 问题 3：可观测性

AI Agent 是黑盒运行的，人类无法实时知道 Agent 做了什么操作。出了问题也难以追溯。

**解决方案：全量操作日志 + 实时仪表盘**
- 每次工具调用都记录：时间、工具名、参数、安全检查结果、耗时、交易哈希
- React 仪表盘实时展示：余额、交易历史、Agent 操作日志、安全策略配置

## 四、系统架构

```
┌──────────────────────────────────────────────┐
│           人类用户 / 浏览器                    │
│         React Dashboard (Vite)                │
│   余额 │ 交易历史 │ 安全配置 │ Agent 日志      │
└──────────────┬───────────────────────────────┘
               │ HTTP API
               ▼
┌──────────────────────────────────────────────┐
│              FastAPI Server                   │
└──────────────┬───────────────────────────────┘
               │ 共享 Wallet Core
┌──────────────┴───────────────────────────────┐
│              MCP Server                       │
│   AI Agent 通过 MCP 协议调用以下工具：         │
│   create_wallet │ get_balance │ send_eth      │
│   sign_message  │ get_transaction │ ...       │
└──────────────┬───────────────────────────────┘
               │
        ┌──────▼──────┐
        │ Security    │  限额检查 / 频率限制
        │ Layer       │  白名单校验 / 操作审计
        └──────┬──────┘
               │
        ┌──────▼──────┐
        │ Wallet Core │  密钥管理 / 交易构建 / 签名
        └──────┬──────┘
               │ JSON-RPC
        ┌──────▼──────┐
        │  Ethereum   │
        │  Sepolia    │
        └─────────────┘
```

**技术栈：**
- 后端：Python + FastAPI + web3.py + MCP SDK
- 前端：React 18 + TypeScript + Vite
- 加密：PBKDF2 + AES-256 (Fernet)
- 区块链：以太坊 Sepolia 测试网

## 五、功能清单

### MCP 工具（AI Agent 调用）

| 工具 | 功能 |
|------|------|
| `create_wallet` | 创建新钱包，私钥加密存储，不暴露给 Agent |
| `import_wallet` | 导入已有私钥 |
| `get_wallet_info` | 查询地址、余额、链状态 |
| `get_balance` | 查询任意地址余额 |
| `send_eth` | 转账（受安全策略约束） |
| `get_transaction` | 查询交易详情 |
| `get_transaction_history` | 查询历史交易 |
| `sign_message` | 对消息签名，证明地址所有权 |
| `set_spending_limit` | 修改支出限额 |

### 使用场景

1. **自动支付链上服务费用**：Agent 判断需要支付时，自动在限额内完成转账
2. **链上数据查询**：Agent 查询余额、交易记录用于分析决策
3. **Agent 身份验证**：Agent 签名消息，向第三方服务证明"我就是这个地址的控制者"
4. **跨 Agent 协作转账**：多个 Agent 协作时，资金在白名单地址间流转

## 六、实际演示

以下是一个真实的多步 Agent 执行过程，展示 AI Agent 如何理解复杂指令并正确拆解执行：

**指令：** "查一下当前钱包余额，然后签名一条消息证明这个地址是我的"

**Agent 执行过程：**

第一步，Agent 自主判断需要先调用 `get_wallet_info`：
```
地址：0x4859F55749b4d2C862aA7e46e4d8AB388e86660c
余额：0 ETH
网络：Sepolia 已连接，区块 #10695955
```

第二步，Agent 调用 `sign_message`：
```
消息：I am the owner of this wallet
签名：0xe24f0f207b3c...1b
Signer：0x4859...660c  ← 与钱包地址完全一致
```

这个过程体现了 AI Agent 的价值：人类只给了一句话的意图，Agent 自主拆解成两步工具调用，判断执行顺序，并将结果关联起来返回。

## 七、与普通钱包的核心区别

| 维度 | 普通钱包（MetaMask） | AI Agent 钱包 |
|------|---------------------|--------------|
| 使用方式 | 人类点击 UI 确认 | 程序接口调用（MCP） |
| 身份验证 | 密码/指纹 | 私钥加密 + 签名隔离 |
| 决策边界 | 无，人类全权控制 | 策略引擎，三级授权 |
| 可观测性 | 仅链上记录 | 完整操作日志 + 仪表盘 |
| 适用场景 | 人类手动操作 | AI 自动化执行 |

## 八、项目定位与边界

本项目是一个**概念验证（PoC）**，完整跑通了"AI Agent + 加密钱包"方向的核心设计：

**做了什么：**
- 明确了双层用户模型和三个核心设计问题
- 实现了私钥安全存储、分级安全策略、全量操作日志
- 通过 MCP 协议与 Claude Code 真实集成并验证调用
- 提供 React 仪表盘供人类监控管理
- 18 个单元测试全部通过

**企业级落地还差什么：**
- 私钥管理：本地文件 → HSM/KMS
- Agent 层：Claude Code API → 私有化部署模型
- 钱包能力：ETH 转账 → 合约调用、多链支持
- 合规：添加 KYC/AML、监管报告能力

核心价值在于：把这个方向的关键问题和架构思路跑通，证明 AI Agent 可以安全、可控地操作区块链钱包。
