# AI 协作过程文档

## 1. 概述

本项目采用 **"人类主导设计 + AI 辅助实现"** 的协作模式，使用 Claude Code 作为编程助手。整个过程中，我负责产品定义、架构决策和质量把控，AI 负责技术调研和代码生成。以下详细记录协作过程。

## 2. 协作阶段

### 阶段 1：自主调研与知识构建

项目启动前，我先独立调研了 Web3 和加密货币钱包的核心概念：

- 阅读了以太坊官方文档，理解了 EOA（外部拥有账户）和合约账户的区别
- 研究了 MetaMask 和cobo等主流钱包的架构。
- 分析了现有 AI Agent 框架（LangChain Tools、MCP）中工具调用的接口范式

在此基础上，与 Claude Code 进行了针对性讨论，用于验证我的理解并补充技术细节。

### 阶段 2：产品定义与需求分析

这是我投入最多思考的阶段：

1. **用户画像的重新定义**
   - AI 最初的方案将 AI Agent 作为唯一用户对象，这存在明显的产品思维缺陷
   - 我指出：钱包的本质是资产管理工具，真正的用户是管理资产的人，Agent 只是操作代理
   - 最终我主导确立了**双层用户模型**：人类用户（Owner）+ AI Agent（Operator），这个模型贯穿了整个架构设计
   - 进一步细分了三类人类用户画像，每类对应不同的安全策略需求

2. **核心问题提炼**
   - 我从实际业务场景出发，提出了 Agent 钱包的三个核心矛盾：
     - "Agent 需要签名能力"与"私钥不能暴露"之间的矛盾 → 签名隔离架构
     - "Agent 需要自主权"与"无限制自主权危险"之间的矛盾 → 分级授权策略
     - "Agent 黑盒运行"与"人类需要掌控"之间的矛盾 → 可观测性体系
   - AI 根据我的问题框架展开了具体的技术方案设计

3. **前端监控的补充**
   - AI 最初只设计了 MCP Server 后端，缺少人类交互界面
   - 我意识到仅有 CLI/API 不够——"人类用户需要可视化的监控手段"，这直接源于我定义的双层用户模型
   - 据此提出增加 React Dashboard，涵盖余额展示、交易历史、安全策略配置和 Agent 操作日志

### 阶段 3：代码实现与 AI 协作

在架构确定后，AI 负责代码实现：

1. **核心模块实现**：AI 按照我的架构设计实现了 config / core / security 三个模块
2. **MCP Server**：AI 实现了多个个工具，我审查了工具命名和参数设计
3. **前端组件**：AI 实现了React 组件，我指定了暗色主题和仪表盘布局
4. **测试**：AI 编写了 18 个单元测试，覆盖密钥派生、钱包操作和安全策略

## 3. 人类与 AI 的价值分工

### 我的核心贡献

| 贡献 | 具体内容 |
|------|----------|
| **产品洞察** | 识别出"双层用户"模型，纠正了 AI 的用户画像偏差 |
| **问题抽象** | 从业务场景提炼出三个核心矛盾，形成设计指导框架 |
| **架构决策** | 确定分层架构、技术选型、三态安全检查等关键设计 |
| **需求补全** | 发现前端监控缺失，从用户视角补充可视化需求 |

### AI 的核心贡献

| 贡献 | 具体内容 |
|------|----------|
| **快速实现** | 在短时间内将架构设计转化为可运行代码 |
| **全栈执行** | 同时处理后端（Python）、前端（React）、文档 |
| **技术细节** | 处理 web3.py API、Fernet 加密、MCP 协议等底层细节 |
| **测试覆盖** | 自动生成 18 个单元测试，覆盖核心逻辑路径 |

## 4. AI 的局限性反思

| 局限 | 分析 |
|------|------|
| **缺乏产品思维** | AI 从技术视角出发，容易忽略用户体验和业务合理性 |
| **假设不透明** | AI 做了"Agent 是唯一用户"的隐含假设但没有说明，需要人类主动质疑 |
| **工程环境** | 虚拟环境管理上遇到困难，说明 AI 在复杂工程环境中的适应能力有限 |
| **无法端到端验证** | 本地只能 mock 测试，无法真实验证链上交互和 Gas 估算 |

## 5. 协作模式总结

```
┌─────────────────────────────────────────────────────┐
│                   人类（我）                          │
│  调研学习 → 产品定义 → 架构决策 → 验收  │
│                  ↕ 协作 ↕                            │
│                    AI                                │
│  技术补充 → 方案细化 → 代码实现 → 测试编写 → Bug 修复  │
└─────────────────────────────────────────────────────┘
```

核心结论：**AI 是强大的执行加速器，但不能替代人类的产品判断和架构决策能力。** 本项目中最有价值的贡献——双层用户模型、三个核心问题的抽象、前端监控需求——都来自人类的思考。AI 的价值在于将这些思考快速、高质量地转化为代码实现，将原本需要数天的开发周期压缩到数小时。
